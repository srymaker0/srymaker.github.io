InnoDB的RC和RR隔离级别的实现

一些MySQL事务基础知识，请移步mysql之基础篇——事务

在MySQL 中的事务是由存储引擎实现的，主要理解InnoDB的MVCC机制，因为这个知识我理解了半天才搞懂，以至于在这里记录下

隔离级别（隔离性）

未提交读（ READ UNCOMMITTED/RU）

⭐已提交读（ READ COMMITTED/RC）

  • 读取已经提交的，读到的数据是最新的（幻读）

⭐可重复读（ REPEATABLE READ/RR）

  • 可以解决幻读
  • 隔离级别高，两事务之间操作相互不影响

  • 被称为可重复读的原因：A事务读到的依旧是之前查到的值，好像忽略了B事务对数据的修改，重复读了自己的值

串行化（SERIALIZABLE）

隔离级别的修改

//查看当前事务级别： 
SELECT @@tx_isolation; 

//设置read uncommitted级别： 
set session transaction isolation level read uncommitted;

//设置read committed级别： 
set session transaction isolation level read committed;

//设置repeatable read级别：
set session transaction isolation level repeatable read;

//设置serializable级别： 
set session transaction isolation level serializable;

LBCC和MVCC

两个事务针对同一数据都发生修改操作时，会存在丢失更新问题。

以至于我们必须在事务对数据进行操作是加上锁

❗若不加锁，则会引发非常严重的问题

LBCC

Lock Based Concurrency Control——基于锁的并发控制

  • 虽然解决了数据安全性问题，但是考虑到真实场景中（有几千万条的数据），这样子加锁，也就意味着当前数据只允许一个事务进行操作，此时其他事务想对该数据操作只能等待，大大降低性能
  • 特点：读写都加锁，所以读读不冲突，读写会冲突，写写更会冲突（死锁问题）

MVCC机制（RR和RC）

Multi Version Concurrency——多版本的并发控制

MVCC使得数据库读不会对数据加锁，普通的SELECT请求不会加锁，提高了数据库的 并发处理能力。

借助MVCC，数据库可以实现READ COMMITTED，REPEATABLE READ等隔离级别，用户可以查看当前数据的前一个或者前几个历史版本，保证了ACID中的I特性 （隔离性)。

特点：读不加锁，读写不冲突

InnoDB中MVCC的实现

MVCC 是用于数据库提供并发访问控制的并发控制技术

由于读不加锁，读写不冲突的特性，提高了系统的并发性能，几乎所有的RDBMS都支持了MVCC

其核心理念就是数据快照，不同的事务访问不同版本的数据快照，从而实现不同的事务隔离级别。

⭐MVCC在mysql中的实现依赖的是undo log 与 read view

Undo log

insert undo log

  • insert undo log是指在insert操作中产生的undo log。由于insert操作的记录，只是对本事务可见，其他事务不可见，所以undo log可以在事务提交后直接删除，而不需要purge操作。

update undo log

  • update undo log是指在delete和update操作中产生的undo log。该undo log会被后续用 于MVCC当中，因此不能提交的时候删除。提交后会放入undo log的链表，等待purge 线程进行最后的删除。

ReadView

使用 READ COMMITTED 和 REPEATABLE READ 隔离级别的事务来说，就需要用到我们上边所说的版本链了。

核心问题就是需要判断一下版本链中的哪个版本是当前事务可见的。

所以设计 InnoD B 的设计者提出了一个ReadView的概念，这个 ReadView 中主要包含当前系统中还有哪些 活跃的读写事务，把它们的事务id放到一个列表中，我们把这个列表命名为为m_ids，并确定三个变量的值：

m_up_limit_id: m_ids事务列表中的最小事务id，如果当前列表为空那么就等于 m_low_limit_id。事务id的下限。

m_low_limit_id ：系统中将要产生的下一个事务id的值。 事务id的上限。

m_creator_trx_id：当前事务id，m_ids中不包含当前事务id。

事务id

  • 事务执行过程中，只有在第一次真正修改记录时（比如使用INSERT、 DELETE、UPDATE语句），才会被分配一个单独的事务id，这个事务id是递增的。

对于RC来说

  • 每次读取数据前都生成一个ReadView

比方说现在系统里有两个id分别为100、200的事务在执行：

# Transaction 100
BEGIN;

UPDATE t SET c = '关羽' WHERE id = 1;

UPDATE t SET c = '张飞' WHERE id = 1;

注意Transaction 100并没有提交

# Transaction 200 
BEGIN;

# 更新了一些别的表的记录 
...

Transaction 200 等等会进行一些操作

  • 活跃的事务：没有提交的事务，还活着

  • 刘备是最初的数据，事务id（trx_id）是80，是最老的数据。接着事物100更新了id=1的数据

假设现在有一个使用READ COMMITTED隔离级别的事务开始执行：

# 使用READ COMMITTED隔离级别的事务 
BEGIN;

# SELECT1：Transaction 100、200未提交 
SELECT * FROM t WHERE id = 1; # 得到的列c的值为'刘备'
  • 此时的这个事务的并不会产生事务id，只有第一次发生修改的时候才会产生。☝

分析：

  • 之前我们说对于RC级别来讲每次查询操作都会返回最新的数据

  • RC级别在读取数据前会生成一个ReadView，此时我们正在干这个事（select读数据），接着产生一个ReadView（存放活跃的事务），范围在【100，200】之间

  • 事务100并没有提交，所以读到的数据还是原来的刘备

我们把事务id为100的事务提交一下，就像这样：

# Transaction 100 
BEGIN;

UPDATE t SET c = '关羽' WHERE id = 1;

UPDATE t SET c = '张飞' WHERE id = 1;

COMMIT;

然后再到事务id为200的事务中更新一下表中id为1的记录：

# Transaction 200 BEGIN;

# 更新了一些别的表的记录 ...

UPDATE t SET c = '赵云' WHERE id = 1;

UPDATE t SET c = '诸葛亮' WHERE id = 1;

此刻，表中id为1的记录的版本链就长这样：

然后再到刚才使用READ COMMITTED隔离级别的事务中继续查找这个id为1的记录，如下：

# 使用READ COMMITTED隔离级别的事务 
BEGIN;

# SELECT1：Transaction 100、200均未提交 
SELECT * FROM t WHERE id = 1; # 得到的列c的值为'刘备'

# SELECT2：Transaction 100提交， Transaction 200未提交 
SELECT * FROM t WHERE id = 1; # 得到的列c的值为'张飞'

这就是RC机制——在每次读数据前产生一个ReadView来确定应该读取哪个版本

  • Transaction 100、200均未提交（说明是活跃的，存在在ReadView中）

    ——ReadView【100，200】，刘备的版本号<100, 则选择刘备，因为可以这样认为，事务id=80是最新的，RC应该返回它给用户

  • Transaction 100提交， Transaction 200未提交（100已提交，因此不会存在在ReadView中）

    ——ReadView【200，200】，张飞和关羽的事物id都小于200，说明对于当前事物这两条记录都可以被看见，但返回第一个看到的版本，也就是张飞，因为其是最新的数据

至此，通过ReadView的产生时机，实现了RC这种隔离级别

对于RR来说

在事务开始后第一次读取数据时生成一个ReadView

针对RC的那个例子来讲，如下:

# 使用REPEATABLE READ隔离级别的事务 
BEGIN;

# SELECT1：Transaction 100、200均未提交 
SELECT * FROM t WHERE id = 1; # 得到的列c的值为'刘备'

# SELECT2：Transaction 100提交， Transaction 200未提交 
SELECT * FROM t WHERE id = 1; # 得到的列c的值仍为'刘备'

分析：

因为在事务开始后第一次读取数据时生成一个ReadView

  • Transaction 100、200均未提交 ——我们生成的ReadView【100，200】，此时做select操作，在版本链中应该找到事务id<100的事务，也就是刘备这条记录
  • Transaction 100提交， Transaction 200未提交 ——生成的ReadView还是【100，200】，如上操作的话，查找到的记录还是刘备

⭐RC和RR隔离级别下，形成的ReadView的时机以及内容是不⼀样的。这就导致了不同隔离级别下同 样的案例，结果是不⼀样的

MVCC下的读操作

当前读

快照读

快照读也就是一致性非锁定读(consistent nonlocking read)是指InnoDB存储引擎通过多版本控制(MVCC)读取当前数据库中行数据的方式。

如果读取的行正在执行DELETE或UPDATE操作，这时读取操作不会因此去等待行上锁的释放。相反地，InnoDB会去读取行的一个最新可见快照。

Tips

  • 如果某个版本的数据对当前事务不可见的话，那就顺着版本链找到下一个版本的数据，继续按照上边的步骤判断可见性，依此类推，直到版本链中的最后一个版本，如果最后一个版本也不可见的话，那么就意味着该条记录对该事务不可见，查询结果就不包含该记录。
  • 在 MySQL 中，READ COMMITTED和REPEATABLE READ 它们生成ReadView的时机不同
  • 所谓的MVCC（Multi-Version Concurrency Control ，多 版本并发控制）指的就是在使用 READ COMMITTD 、 REPEATABLE READ 这两种隔离级别的事 务在执行普通的 SEELCT 操作时访问记录的版本链的过程，这样子可以使不同事务的 读写 、 写-读 操作并发执行，从而提升系统性能。
